# 分层模型版本管理设计

## 目标

将前端模型名称管理改为 `class → model → version` 三层结构，同时保持后端结果文件目录仍使用单层 `class_model_version` 名称，不引入新的模型元数据表。

本次改动覆盖：

- 上传结果时分开输入 class、model 和 version。
- class 与 model 支持从已有数据逐层匹配，也支持输入新值。
- 新 class 或新 model 在上传前要求用户确认。
- 总览筛选增加 class 和 model 下拉菜单。
- 上传测评集与上传结果图的配置项改为两行布局。
- 仓库内 mock 模型目录迁移为三段式名称。
- 兼容无法按三段式解析的历史真实模型名称。

## 命名与兼容规则

### 标准名称

标准模型完整名称由后端唯一拼接：

```text
class_model_version
```

class、model、version 均为必填字段，并且：

- 去除首尾空白后不能为空。
- 不允许包含下划线 `_`。
- 继续遵守现有存储组件安全规则，不允许路径分隔符、`.`、`..` 等不安全值。

前端提供即时校验和明确提示，后端执行相同约束作为最终安全边界。前端不直接提交完整目录名，避免绕过分字段校验。

### 历史名称

只有恰好能按 `_` 分成三个非空部分的名称才视为标准三段式名称。

无法解析的历史名称返回：

```json
{
  "class_name": null,
  "model_name": null,
  "version": "原完整名称",
  "full_name": "原完整名称"
}
```

历史目录和数据库记录不做批量改名。它们继续参与对战、统计和“全部”筛选，但不生成空白的 class 或 model 下拉选项。

### Mock 数据

仓库自带 mock 模型统一迁移为：

```text
test_<原模型名>_default
```

例如：

- `Atlas` → `test_Atlas_default`
- `Beacon` → `test_Beacon_default`
- `Cipher` → `test_Cipher_default`
- `Prism` → `test_Prism_default`
- `Mosaic` → `test_Mosaic_default`

同步更新引用这些名称的仓库测试、fixture、脚本和文档。运行时数据库中的用户数据不随 mock 数据迁移。

## 模型目录数据源

不新增模型元数据表。后端为每个任务类型动态建立模型目录，合并以下来源：

1. 结果文件根目录中的模型版本目录。
2. 数据库 `pair_tasks.v_a/v_b` 中出现过的名称。
3. 数据库 `results_log.v_a/v_b` 中出现过的名称。

合并后按完整名称去重并排序，再套用标准名称或历史名称解析规则。

选择“目录 + 数据库”作为数据源可以同时覆盖：

- 已上传但还没有产生评测记录的模型。
- 已产生历史评测记录但对应文件目录当前缺失的模型。

模型目录接口按任务类型返回结构化条目：

```json
{
  "task_type": "T2I",
  "models": [
    {
      "class_name": "test",
      "model_name": "Atlas",
      "version": "default",
      "full_name": "test_Atlas_default"
    }
  ]
}
```

## 上传结果交互

上传新结果图的第一行依次为：

1. 任务类型
2. class
3. model
4. version

第二行依次为：

1. 场景
2. 结果 zip
3. 上传按钮

### 级联输入

class 和 model 使用支持“下拉匹配 + 自由输入”的控件：

- class 候选来自当前任务类型的全部标准模型条目。
- 选择或输入 class 后，model 候选只显示该 class 下已有的 model。
- 切换任务类型时重新加载目录并清理不再匹配的选择。
- version 可自由输入；选定 class 和 model 后，可提供该组合下已有 version 作为匹配候选。

浏览器原生 `datalist` 可满足下拉匹配和自由输入需求，并保持当前单文件前端结构；不增加第三方组件依赖。

### 新建确认

提交前根据当前模型目录判断：

- class 不存在时，确认“class `<值>` 尚不存在，是否新建？”
- class 已存在但该 class 下的 model 不存在时，确认“model `<值>` 尚不存在，是否在 class `<值>` 下新建？”
- class 和 model 都存在时不提示。
- version 无论是否存在都不触发“新建”确认。

如果 class 和 model 都是新的，先确认 class，再确认 model。任一确认取消都中止上传、保留已输入内容并聚焦对应输入框。

ZIP 文件名自动改名确认仍沿用现有流程。模型新建确认发生在发起上传请求之前；ZIP 自动改名确认发生在后端校验返回之后，两者职责独立。

### 后端上传接口

`POST /api/upload` 改为接收：

- `task_type`
- `class_name`
- `model_name`
- `version`
- `scene`
- `file`
- `auto_rename`

后端分别校验三个名称字段，再拼接完整名称，并继续调用现有结果 ZIP 校验和文件保存流程。最终目录仍为：

```text
<result_root>/<class>_<model>_<version>/<scene>/
```

响应中可返回 `full_name`，供前端刷新目录和展示成功消息。接口不接受前端提供的完整名称作为可信路径值。

## 总览筛选

现有总览筛选中增加 class 和 model 下拉菜单：

- class 默认值为“全部 class”。
- model 默认值为“全部 model”。
- 选择 class 后，model 选项收窄为该 class 下已有 model。
- 更换 class 时，如果当前 model 不属于新 class，则重置为“全部 model”。

筛选一个模型对时，只要任一对战方满足已选 class/model 条件，该模型对就保留。class 和 model 同时选择时，必须由同一个对战方同时满足，不能由 A 方满足 class、B 方满足 model。

旧式 null 模型：

- 在两个筛选都为“全部”时正常显示。
- 选择具体 class 或 model 后不匹配。
- 不在 class/model 下拉菜单中生成空白选项。

现有模型关键字、场景及其他筛选条件继续与 class/model 条件做 AND 组合。

## 上传区域布局

### 上传新测评集

第一行：

1. 任务类型
2. 场景名

第二行：

1. Prompt txt
2. 参考图 zip
3. 上传按钮

T2I 下参考图字段保持隐藏；布局应让剩余字段自然占用可用宽度。

### 上传新结果图

第一行：

1. 任务类型
2. class
3. model
4. version

第二行：

1. 场景
2. 结果 zip
3. 上传按钮

两个 block 在窄屏下继续使用现有响应式规则退化为单列，避免固定两行导致横向溢出。

## 模块边界

### 存储层

负责：

- 单字段名称校验。
- 三字段完整名称拼接。
- 完整名称解析。
- 扫描结果目录中的模型名称。
- 使用完整名称保存上传文件。

### 数据库查询层

负责读取 `pair_tasks` 和 `results_log` 中去重后的模型名称，不负责解释文件路径。

### 模型目录服务

负责合并目录和数据库名称、解析、去重、排序，并为 API 生成结构化响应。

### 看板前端

负责：

- 加载模型目录。
- class/model/version 级联候选。
- 新 class/model 确认。
- 客户端即时校验。
- 总览 class/model 筛选。
- 两行表单布局。

## 错误处理

- 任一字段为空、含 `_` 或不符合安全存储规则时，后端返回明确的字段级错误。
- 模型目录加载失败时，上传输入仍允许自由输入，但显示目录加载失败提示；提交仍由后端校验。
- 用户取消新建确认时不清空表单和文件选择。
- 上传成功后刷新模型目录和总览；刷新失败不把已经成功的上传误报为失败。
- 历史名称解析失败不是错误，按 null 兼容结构返回。

## 测试策略

遵循 TDD，先增加失败测试，再实现最小代码。

后端测试覆盖：

- 标准三段名称解析。
- 非标准历史名称返回 null 兼容结构。
- 三字段拼接。
- 空值、下划线和路径字符校验。
- 结果目录与两个数据库表的名称合并、去重和排序。
- 上传接口使用后端拼接的完整名称写入配置目录。
- 旧上传方式不能绕过分字段校验。

前端测试覆盖：

- 上传表单包含 class、model、version 三个字段。
- class/model/version 候选按任务类型和上级选择级联。
- 新 class 与新 model 的确认顺序及取消行为。
- 非法下划线在发请求前被拦截。
- FormData 提交分字段名称，不提交可信完整名称。
- class/model 总览筛选由同一个对战方匹配。
- null 历史模型不会产生空白筛选项。
- 两个上传 block 均使用两行桌面布局并保持窄屏单列。
- 动态选项继续使用安全 DOM API，不插入后端提供的 HTML。

回归验证覆盖：

- 全量 Python 测试。
- 前端 Node 辅助测试。
- mock 目录改名后，任务生成、图片 URL、评测流程、总览、排行榜和导出仍可使用完整模型名称。

## 完成标准

- 新上传结果仅能生成合法的 `class_model_version` 目录。
- class/model 支持已有值匹配和新值确认。
- 总览能按 class 和 model 快速筛选。
- 历史非三段名称无需迁移即可继续展示和参与统计。
- mock 数据统一显示为 `test / 原模型名 / default`。
- 上传区域在桌面端按要求分为两行，在窄屏无横向溢出。
- 所有新增测试和现有回归测试通过。
