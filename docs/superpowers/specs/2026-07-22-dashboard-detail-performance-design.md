# 看板明细图片性能优化设计

## 目标

解决看板明细包含数百张 2K 图片时，打开弹窗、筛选、滚动和关闭后页面卡顿及内存占用过高的问题。优化后，列表仅加载当前页的小尺寸缩略图，点击缩略图仍使用现有高清原图预览。

## 已确认范围

- 明细列表每页显示 50 条，切页时只创建当前页 DOM。
- 列表图片使用最长边 256px 的 WebP 缩略图；点击后继续加载原始图片。
- 缩略图按请求即时生成并落盘缓存，后续请求直接复用缓存文件。
- 列表图片启用原生懒加载、异步解码和低请求优先级。
- 文件名筛选使用 200ms 防抖，并在筛选条件改变后回到第一页。
- 关闭明细弹窗时清空表格、分页状态和明细数据，释放图片节点。
- 保持现有明细筛选、评测员筛选、判定展示和高清预览行为不变。
- 不优化坏例列表，不引入第三方前端依赖，不改变原图文件。

## 方案比较

### 方案 A：仅增加 `loading="lazy"`

改动最小，但用户滚动后仍会下载和解码全部 2K 原图，也无法解决筛选时重建数百行 DOM 的问题。只适合作为临时缓解，不单独采用。

### 方案 B：客户端分页加原图懒加载

能限制同时存在的 DOM 数量，但当前页仍需解码 100 到 150 张原图，网络和内存成本仍与实际 76px 展示尺寸不匹配。

### 方案 C：缩略图缓存、分页、懒加载和关闭清理

同时限制传输体积、图片解码尺寸和 DOM 数量，且不改变高清预览。采用此方案。

## 后端设计

新增缩略图路由：

```text
GET /api/image-thumbnail?kind=result&task_type=T2I&model=...&scene=...&filename=...
GET /api/image-thumbnail?kind=ref&task_type=TI2I&scene=...&filename=...
```

路由通过现有存储路径解析函数定位源图片，不接受任意文件路径。`kind=result` 必须提供模型名称；`kind=ref` 忽略模型名称。找不到源图时返回 404，不支持的图片或解码失败时返回可读的 400 系列错误。

新增独立的缩略图服务模块，负责：

1. 解析并校验源图片位于允许的数据目录中。
2. 根据图片种类、任务类型、模型、场景、文件名和源文件修改时间生成稳定缓存键。
3. 使用 Pillow 按比例缩放到最长边不超过 256px，并输出 WebP。
4. 先写临时文件，再原子替换为最终缓存文件，避免并发请求读取半成品。
5. 缓存命中时不重新解码原图。

缓存目录位于应用数据目录下的专用 `.thumbnails` 子目录，并加入 Git 忽略。响应携带浏览器缓存头；源文件修改后 URL 对应的后端缓存键变化，避免继续复用旧缩略图。

本次不修改 `/api/detail_results` 的响应结构。数据量为数百条时，JSON 和文本处理成本远小于图片解码；客户端分页足以限制当前瓶颈，也避免把现有多选评测员筛选逻辑迁移到后端。

## 前端设计

明细状态新增当前页和固定页大小。`renderDetailTable()` 先执行现有筛选，再计算总页数并对结果切片，只为当前 50 条记录创建表格行。

表格中的图片地址由原始 `imageUrl()` 改为缩略图 URL：结果图传递模型、场景和文件名，参考图传递任务类型、场景和文件名。图片节点设置：

```javascript
image.loading = "lazy";
image.decoding = "async";
image.fetchPriority = "low";
image.width = 76;
image.height = 76;
```

点击事件仍使用现有 `buildPreviewPayload()` 中的原始图片 URL，因此高清预览不降质。

表格下方新增“上一页 / 第 N 页，共 M 页 / 下一页”。筛选文件名、判定或评测员变化时将页码重置为 1。文件名输入通过 200ms 防抖触发渲染，其他筛选继续即时响应。

关闭 `detail-modal` 时：

- 清空 `#detail-body` 和分页控件；
- 将 `state.detailRows` 设为空数组；
- 清空 `state.selectedDetailWorkers`、`state.currentDetail` 和当前页；
- 递增请求编号，使迟到的明细请求不能重新填充已关闭弹窗。

## 数据流

1. 用户打开明细，前端请求现有明细 JSON。
2. 前端筛选数据并只渲染当前页 50 条。
3. 进入浏览器懒加载范围的图片请求缩略图接口。
4. 后端命中缓存则直接返回 WebP；未命中则从原图生成并缓存后返回。
5. 用户点击缩略图时，现有高清预览加载原图。
6. 用户切页或筛选时，旧行被移除，只保留新一页图片节点。
7. 用户关闭弹窗时，明细 DOM 和状态被清空。

## 错误与边界处理

- 缩略图生成失败时，该图片使用浏览器的失败占位表现，表格其他内容和高清预览入口不受影响。
- 总条数为 0 时显示现有空状态并隐藏分页按钮。
- 当前页超过筛选后的最大页数时自动夹紧到最后一页。
- 快速打开、关闭或切换不同明细时，仅最新请求可以更新状态。
- 源图片已更新时，根据修改时间生成新的缓存文件。
- 缓存目录创建失败或文件不可写时返回明确错误，不修改原始图片。

## 测试方案

- 缩略图服务测试：尺寸不超过 256px、保持比例、输出 WebP、缓存复用、源图修改后刷新缓存、非法路径和缺失文件被拒绝。
- API 测试：结果图和参考图均能返回缩略图，缺失参数和不存在图片返回正确错误。
- 前端契约及 Node 探针：每页最多 50 条、翻页边界、筛选重置页码、图片属性、缩略图 URL、点击载荷仍为原图、关闭释放状态、迟到请求失效。
- 完整 Python 测试套件：确认看板、上传、导出、权限和高清预览无回归。

## 完成标准

- 含数百条记录的明细首次只创建最多 50 行、最多 100 张 T2I 或 150 张 TI2I 列表图片。
- 列表网络请求加载的是最长边不超过 256px 的 WebP，而不是原始 2K 文件。
- 高清预览继续显示原始分辨率图片。
- 搜索输入不会逐按键立即重建表格。
- 关闭明细后不保留列表图片节点或明细数据。
- 新增专项测试和完整测试套件通过。
