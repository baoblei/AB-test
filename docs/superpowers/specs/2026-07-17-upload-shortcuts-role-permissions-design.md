# 上传状态、评测快捷键与三级用户权限设计

## 目标

本次优化包含三项相互独立但共同涉及前后端交互的行为调整：

1. 看板上传状态栏用绿色表示成功、红色表示失败。
2. 评测快捷键始终填写第一个尚未选择的维度，不因已有选择而失效。
3. 用户身份扩展为 `admin`、`manager`、`evaluator` 三级，并在后端统一限制管理、上传和导出权限。

## 角色与权限

系统继续使用 `users.role` 作为单一权限来源，合法角色固定为：

| 身份 | 管理用户 | 上传数据 | 导出数据 | 参与评测和查看看板 |
| --- | --- | --- | --- | --- |
| `admin`（超级管理员） | 是 | 是 | 是 | 是 |
| `manager`（管理员） | 否 | 是 | 是 | 是 |
| `evaluator`（评测员） | 否 | 否 | 否 | 是 |

注册接口显式将新用户保存为 `evaluator`。初始化创建的内置 `admin` 用户仍为最高身份。

认证模块提供职责清晰的权限依赖：用户管理接口继续要求 `admin`；上传与全部导出接口允许 `admin` 和 `manager`。旧版导出接口也必须纳入权限校验，避免绕过新版导出限制。

管理后台为每个用户提供角色下拉选择。服务端只接受上述三个合法值，并执行以下保护：

- admin 不能修改自己的角色。
- 任何修改都不得使系统失去最后一个有效 admin。
- 角色变更写入操作日志。
- 用户启停保持现有功能；禁用最后一个有效 admin 同样被拒绝。

## 界面交互

### 上传状态

看板保留所有上传入口，不根据当前用户角色隐藏功能。上传请求成功后，公共上传状态栏显示绿色文字；请求失败后显示红色文字。评测员上传时，后端返回 403，状态栏显示：

> 当前用户没有权限，请联系管理员

状态样式通过语义化成功/失败 class 切换，开始新请求时清除上一次状态，避免残留颜色误导用户。

### 导出状态

看板保留导出按钮和弹窗。评测员可以打开弹窗，但获取导出选项、预览或下载时后端拒绝请求，导出弹窗状态栏以红字显示：

> 当前用户没有权限，请联系管理员

前端不作为权限安全边界；所有真实限制均由后端实施。

### 评测快捷键

每次收到 `1`、`2`、`3` 快捷键时，按当前页面维度顺序查找第一个尚未选择的维度，并分别选择左胜、平局、右胜。鼠标或快捷键已经填写的维度会被跳过。

当所有维度均已选择时，再按数字键不覆盖已有选择。`Enter` 提交规则保持不变。切换任务后不依赖历史索引，而是根据当前表单实时判断空白维度，从而保证行为一致。

## 数据流与错误处理

角色修改流程为：管理后台提交目标角色 → admin 权限校验 → 合法角色及自修改/最后 admin 保护 → 数据库更新 → 操作日志 → 刷新用户列表。

上传和导出流程为：浏览器发起现有请求 → 后端角色依赖校验 → 有权限则执行原业务逻辑，无权限则返回 403 和统一中文提示 → 页面状态栏以红色渲染该提示。

权限校验失败不得执行文件写入、导出文件生成或数据修改。

## 测试与验收

采用测试先行方式覆盖：

- 新注册用户角色为 `evaluator`。
- `admin`、`manager`、`evaluator` 的管理、上传和导出权限矩阵。
- 非法角色、自身降级、禁用或降级最后一个有效 admin 均被拒绝。
- 管理后台包含三级角色选择并能调用角色更新接口。
- 上传成功状态使用绿色样式，失败及无权限状态使用红色样式。
- 导出无权限错误在弹窗状态栏以红色显示。
- 快捷键跳过已有选择，从第一个空白维度继续；全部已选时不覆盖。
- 现有测试套件继续通过。

## 非目标

本次不引入逐项权限开关，不隐藏评测员的上传或导出入口，不改变评测任务分配逻辑，也不新增角色创建功能。
