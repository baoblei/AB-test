# 看板评测结果导出与北京时间统一设计

日期：2026-07-15

## 1. 背景与目标

当前看板仅提供返回 JSON/CSV 文本的简易导出接口，无法按场景、维度和评测人精确筛选，也不能生成带图片的归档文件。同时，SQLite 使用 `CURRENT_TIMESTAMP` 写入 UTC 无时区时间，而页面直接展示原值，导致业务时间比北京时间慢 8 小时；评测耗时还包含任务与图片加载时间。

本次改造目标：

- 在每个模型对的总 PK 折叠总览中提供独立导出入口。
- 支持按场景、评测维度、评测人、北京时间范围、评测模式、判定结果和坏例状态筛选。
- 生成包含 Overall 总览和各维度明细的 Excel。
- 可选打包参与所选结果的模型图片；TI2I 同时打包参考图。
- 分别适配 T2I 与 TI2I 的维度、参考图和目录结构。
- 将项目业务时间统一为带 `+08:00` 的北京时间，并修正评测耗时起点。

不在本次范围内：后台异步导出队列、跨模型对批量导出、在 Excel 单元格中嵌入图片，以及修改 JWT 等协议时间。

## 2. 已确认决策

- 采用服务端同步生成文件的方案。
- 未勾选图片时直接下载 XLSX；勾选图片时下载包含 XLSX 和图片目录的 ZIP。
- 同一图片被多名评测人评价时，明细保留多行，图片只打包一次。
- Overall Sheet 是汇总表；各评测维度各自使用一个明细 Sheet。
- 所有业务时间统一保存为北京时间 ISO 8601，例如 `2026-07-15T09:30:00+08:00`。
- 历史 UTC 无时区时间执行一次幂等迁移，增加 8 小时并补充 `+08:00`。
- 图片缺失不会中断导出，明细中明确标记缺失状态。

## 3. 看板交互

### 3.1 导出入口

在每个模型对的总 PK 标题栏操作区增加“导出”按钮，与“统计”和“坏例详情”并列。按钮固定当前看板任务类型和模型对，避免在弹窗中再次选择模型。

### 3.2 导出配置弹窗

弹窗展示以下信息和选项：

- 固定信息：任务类型、模型 A、模型 B。
- 场景：多选，默认全选当前模型对已有数据的场景。
- 明细维度：多选；T2I 为美学、合理性、一致性，TI2I 额外包含保真度。Overall 总览固定生成，不受此多选项影响。
- 评测人：多选，默认全选当前模型对已有数据的评测人。
- 时间范围：北京时间起止值，默认覆盖当前模型对的全部记录。
- 评测模式：全部、多维度评测、整体快速评测。
- 判定结果：全部、A 胜、平局、B 胜。
- 坏例范围：全部、存在坏例、不存在坏例。“存在坏例”表示 A 或 B 至少一侧含坏例标签。
- 导出图片：默认关闭。
- 包含坏例字段：默认开启。
- 包含评测耗时：默认开启。

弹窗底部实时显示当前筛选预计包含的评测记录数。筛选结果为空时禁用导出并显示原因。

### 3.3 筛选语义

场景、评测人、时间、评测模式和坏例范围是全局行筛选。判定结果按 Sheet 独立应用：Overall 使用 `overall` 判定，各维度 Sheet 使用该维度的判定。

整体快速评测记录只参与 Overall Sheet。非整体维度 Sheet 只包含 `eval_mode=full` 且该维度有有效结果的记录。即使用户只选择整体快速评测，所选维度 Sheet 仍保留表头并显示零条明细。

## 4. Excel 内容

### 4.1 Overall Sheet

Sheet 名固定为 `Overall`。顶部记录：

- 导出生成时间（北京时间）
- 任务类型及模型对
- 场景、维度、评测人、时间、模式、结果和坏例筛选条件
- 是否包含图片、坏例字段和耗时
- 最终评测记录数及缺失图片数量

统计表第一行是全部所选场景合计，后续每行一个场景，包含：

- 场景与样本数
- A 胜、平局、B 胜的数量与比例
- A 相对 B 压制比 `(A胜 + 平) / (B胜 + 平)`
- B 相对 A 压制比 `(B胜 + 平) / (A胜 + 平)`
- A、B 坏例数量与坏例占比

分母为 0 且分子大于 0 时压制比显示 `∞`；分子和分母都为 0 时显示 `-`。

### 4.2 维度明细 Sheet

每个选中维度生成一个 Sheet，名称为 `美学明细`、`合理性明细`、`一致性明细` 或 `保真度明细`。每条评测记录一行，至少包含：

- 任务类型、模型 A、模型 B
- 场景、图片名、Prompt
- 当前维度判定
- 评测人、评测模式
- 评测时间（北京时间）
- 评测耗时（配置开启时）
- A/B 坏例类别和具体标签（配置开启时）
- A/B 图片相对路径与图片状态
- TI2I 参考图相对路径与图片状态

判定值使用模型真实名称或“平局”，不使用内部的 `tie` 文本。列宽、冻结表头、筛选、换行和百分比格式由服务端统一设置。

### 4.3 Prompt 和重复记录

Prompt 按任务类型、场景和不带扩展名的图片名读取，兼容现有旧目录回退规则。同一图片若由多名评测人评价，或者同一评测人产生多条合法记录，每条记录都保留，不做聚合覆盖。

## 5. 图片归档

勾选“导出图片”时，ZIP 结构为：

```text
评测导出_<任务类型>_<模型A>_vs_<模型B>_<北京时间>.zip
├── 评测结果.xlsx
└── images/
    └── <场景名>/
        ├── <模型A>/
        │   └── <图片名>
        ├── <模型B>/
        │   └── <图片名>
        └── ref/
            └── <图片名>  # 仅 TI2I
```

模型结果图路径固定为 `images/<场景名>/<模型名>/<图片名>`；TI2I 参考图路径固定为 `images/<场景名>/ref/<图片名>`。图片使用原始文件名和扩展名。

图片集合是所有通过筛选并参与任一已生成 Sheet 的评测记录之并集；只选择整体快速评测时也会按 Overall 记录打包图片。归档前对场景名、模型名和文件名执行路径安全校验，禁止绝对路径及 `..`。缺失文件不写入 ZIP，对应明细状态标记为“文件不存在”。

## 6. 后端结构与接口

### 6.1 模块边界

- `app_core/export_service.py`：筛选、统计、工作簿构建、图片收集和 ZIP 构建。
- `app_core/time_utils.py`：北京时间生成、解析、格式化和历史时间迁移所需的纯函数。
- `app_core/schemas.py`：导出请求模型及筛选枚举校验。
- `app_core/dashboard_service.py`：保留现有看板查询，不承载文件生成职责。
- `app_core/storage.py`：提供模型结果图和参考图的安全本地路径解析。

使用 `openpyxl` 生成 XLSX，并在 `requirements.txt` 中声明依赖。ZIP 使用 Python 标准库 `zipfile`。文件在临时目录中生成，响应完成后清理，避免将大图片全部保留在进程内存中。

### 6.2 接口

`GET /api/export_options`

参数：`task_type`、`v1`、`v2`。返回规范化后的模型 A/B、可选场景、评测人、维度、最早/最晚北京时间及总记录数。

`POST /api/export/preview`

接收完整筛选配置，返回 Overall 和每个所选维度的预计行数，用于弹窗反馈和空结果校验。

`POST /api/export`

接收完整筛选配置，返回 XLSX 或 ZIP 二进制响应。文件名使用清理后的任务类型、模型名和北京时间。原有 `GET /api/export` JSON/CSV 接口继续保留，避免破坏已有调用。

请求中的模型顺序统一按现有规则排序为 `v_a`、`v_b`，所有 A/B 判定、统计和目录名称均使用该规范顺序。

## 7. 北京时间统一

### 7.1 存储格式

所有业务时间统一为秒级 ISO 8601 北京时间：

```text
YYYY-MM-DDTHH:MM:SS+08:00
```

涉及字段：

- `results_log.timestamp`
- `operation_logs.timestamp`
- `users.created_at`
- `users.last_login`

新数据库的字段默认值使用固定 `+08:00` 规则。现有应用写入路径显式传入 `now_beijing_iso()`，包括用户注册、登录、操作日志、提交评测和跳过任务，避免依赖服务器本地时区。

### 7.2 历史数据迁移

数据库增加迁移元数据记录 `beijing_time_v1`。首次启动时，在单个事务中将上述字段里形如 `YYYY-MM-DD HH:MM:SS` 的历史无时区值视为 UTC，增加 8 小时后写成北京时间 ISO 格式。带 `T` 和 `+08:00` 的值不处理；迁移完成后写入版本标记，因此重复启动不会二次偏移。

迁移前后记录数量、空值和主键保持不变。无法解析的异常时间保留原值并记录诊断信息，不阻塞应用启动。

### 7.3 查询与展示

看板明细、坏例详情、个人历史、管理日志、今日统计和 Excel 均使用北京时间。页面只做 `T` 与时区后缀的展示格式化，不再执行 UTC 与本地时区换算。导出时间范围也直接使用带 `+08:00` 的输入查询。

JWT 过期时间继续使用标准 UTC/Unix 时间戳，不属于业务时间迁移范围。

### 7.4 评测耗时

当前计时器在请求任务前启动，会把接口和图片加载时间计入耗时。调整为：任务数据渲染后等待当前任务所需图片加载完成或失败，再重置并启动计时器。提交时仍记录整数秒；切换任务、跳过任务和页面重新初始化时清理旧计时器，避免并行计时。

## 8. 错误处理

- 无效任务类型、维度、模型对或筛选值返回 422，并给出中文错误信息。
- 当前模型对没有符合条件的评测记录时不生成空文件，返回明确提示。
- Excel 生成失败或临时文件写入失败返回 500，同时确保清理临时目录。
- 单张图片缺失、损坏或无读取权限不会中断 Excel 和其余图片导出，明细记录具体状态。
- 导出按钮在请求期间进入禁用和加载状态，重复点击不会并发发起相同下载。

## 9. 测试策略

实现遵循测试先行，覆盖：

- 时间纯函数生成固定 `+08:00` 格式。
- 历史 UTC 无时区值正确增加 8 小时，迁移幂等且保留空值与异常值。
- 新评测、跳过、用户注册/登录和操作日志直接写入北京时间。
- 今日统计在北京时间午夜边界正确。
- 场景、维度、评测人、时间、模式、结果和坏例筛选可独立及组合工作。
- Overall 合计、分场景胜平负、压制比和坏例占比正确。
- T2I 工作簿不包含保真度和参考图列；TI2I 包含保真度及参考图列。
- 多评测人产生多行明细，同一图片在 ZIP 中去重。
- ZIP 路径严格符合 `场景/模型/图片名` 和 `场景/ref/图片名`。
- 缺图时导出成功且明细标记缺失。
- 前端弹窗默认值、动态维度、预计数量、空数据和下载状态正确。
- 计时器只在任务图片完成加载后启动。

完成后运行 Python 测试、模块编译、模板 JavaScript 语法检查，并使用本地浏览器分别验证 T2I/TI2I 的 XLSX 和 ZIP 下载。

## 10. 验收标准

- 每个模型对总 PK 标题栏均可打开导出配置弹窗。
- 导出选项只来自当前任务类型和模型对，筛选组合与文件内容一致。
- XLSX 始终含 Overall，且每个所选维度有独立明细 Sheet。
- 每条维度明细含图片名、Prompt、判定和评测人；配置开启时含耗时与坏例。
- T2I 图片包包含 A/B，TI2I 图片包包含 A/B/ref，目录结构符合本设计。
- 项目业务时间在数据库、API、界面和 Excel 中均为北京时间 `+08:00`。
- 历史时间只迁移一次，现有记录不会丢失或重复偏移。
- 评测耗时不再包含任务和图片加载等待时间。
